# Plan: Fix VAE Encoder Architecture Mismatch for Proper img2img

## Problem

The img2img mode currently uses a 3→4 channel padding workaround (`load_init_image` in `inference.rs`) instead of proper VAE encoding. The Rust `VideoEncoder` in `ltx-video-vae` has a fundamentally different architecture from the Python LTX-Video VAE encoder.

## Architecture Comparison

### Python VAE Encoder (from checkpoint analysis)
- **180M params, 94 tensors, NO timestep conditioning**
- Input: `space_to_depth(r=4)` converts 3-channel RGB → 48 channels
- `conv_in`: 48 → 128
- 10 heterogeneous `down_blocks` (3 block types):
  - **ResBlock stages** (blocks 0, 3, 6, 8, 9): sequential resblocks, no spatial change
  - **Stride-2 convs** (blocks 1, 4, 7): simple spatial /2 downsample
  - **Channel-change downsamples** (blocks 2, 5): stride-2 + channel doubling + 1×1 shortcut + GroupNorm
- Channel progression: 128 → 128 → 256 → 256 → 512 → 512 → 512 → 512
- Spatial: 5× stride-2 = **32× total downsample**
- `conv_out`: 512 → 129 (distribution parameters: mean + logvar + 1 extra)
- **No mid block, no conv_norm_out**

### Rust VideoEncoder (current, WRONG)
- `space_to_depth(r=2)` — wrong ratio (12 vs 48 input channels)
- `conv_in`: 3 → 128 (wrong, should be 48 → 128)
- Homogeneous down_blocks + `UNetMidBlock3D` (Python has neither)
- `conv_norm_out` before `conv_out` (Python has none)
- Simple `conv_out` to `latent_channels` (no distribution sampling)

## Approach: 3 Parallel Workstreams

### 1. Rewrite `VideoEncoder` to match Python architecture

**File: `crates/ltx-video-vae/src/lib.rs`**

Replace the `VideoEncoder` struct and its `new()`/`forward()` methods:

```
struct VideoEncoder {
    conv_in: Conv3d(48 → 128),
    blocks: Vec<EncoderBlock>,  // 10 blocks with 3 types
    conv_out: Conv3d(512 → 129),  // distribution params
}
```

Three block types as an enum:
```rust
enum EncoderBlock {
    ResBlocks(Vec<ResBlock>),           // blocks 0, 3, 6, 8, 9
    DownsampleConv(Conv3d),             // blocks 1, 4, 7
    ChannelChangeDownsample {           // blocks 2, 5
        conv1: Conv3d,
        conv2: Conv3d,
        shortcut: Conv3d(1×1),
        norm: GroupNorm,
    },
}
```

`forward()` flow:
1. `space_to_depth(x, 4)` — r=4, not r=2
2. `conv_in`
3. Iterate 10 blocks
4. `conv_out` → output 129 channels
5. Split into `mean(64)` + `logvar(64)` + `scale(1)`
6. Reparameterize: `latent = mean + exp(0.5 * logvar) * noise`

**File: `crates/ltx-video-vae/src/encoder_blocks.rs`**

Rewrite to provide the 3 block constructors matching the Python checkpoint.

**File: `crates/ltx-video-vae/src/configurator.rs`**

Update `VideoVAEConfig` and `from_config()` to wire the new encoder with correct channel multipliers and block counts.

**Key config values** (derived from checkpoint):
- `base_channels`: 128
- `channel_multipliers`: not used — explicit block structure
- `conv_in_channels`: 48 (from space_to_depth r=4 of 3 RGB channels)
- `latent_channels`: 64 (from conv_out 129 = 2×64 + 1)
- `spatial_downsample_factor`: 32 (5× stride-2)

### 2. Add VAE encoder weight loading to inference binary

**File: `crates/ltx-core/src/bin/inference.rs`**

- Add `--vae-weights` CLI argument (path to the original `.safetensors` file containing `vae.encoder.*` keys)
- Create a temporary `VarStore` for the VAE encoder (following the text encoder pattern)
- Build encoder via `configurator::from_config()` with correct config
- Load weights with key mapping: Rust var path `encoder.xxx` → checkpoint key `vae.encoder.xxx`
- Run encoding, then drop the VarStore to free memory before transformer denoising

**Weight loading approach**: The existing `load_weights()` function uses 3 fallback strategies (exact, T5 prefix, dots). For VAE, we need a 4th strategy: map `encoder.xxx` → `vae.encoder.xxx` by prepending the `vae.` prefix. Alternatively, create a wrapper that strips `vae.encoder.` from checkpoint keys before loading.

### 3. Replace `load_init_image` workaround with proper VAE encoding

**File: `crates/ltx-core/src/bin/inference.rs`**

Replace `load_init_image()` with `encode_init_image()`:

```
fn encode_init_image(img_path, vae_weights_path, frames, device) -> Tensor {
    1. Load image at ORIGINAL resolution (not latent dims)
    2. Resize to match spatial dims divisible by 32
    3. Normalize to [-1, 1]
    4. Reshape to (1, 3, T, H, W) — replicate frame across T
    5. Build VideoEncoder from config + load VAE weights
    6. Forward through encoder → (1, 64, T', H/32, W/32) latent
    7. Drop encoder (free memory)
    8. Return latent tensor
}
```

Update the denoising initialization:
- Remove the `--height`/`--width` requirement for img2img init (latent dims determined by encoder)
- OR require `--height`/`--width` to match encoder output after downsampling
- The denoising loop itself is unchanged — it works on whatever latent shape the encoder produces

### Latent Channel Mismatch: 64 vs 4

**Critical issue**: The transformer expects 4-channel latents (`c=4` on line 381 of inference.rs), but the VAE encoder produces 64-channel latents. Two options:

**Option A (recommended):** Add a learned projection layer in the VAE encoder's output: `Linear(64 → 4)` that maps to the transformer's latent space. This is common in video diffusion models (e.g., Stable Video Diffusion uses a `quant_conv` to project from VAE latent to transformer latent).

**Option B:** Adjust the transformer's `c` parameter to 64 and update `patch_dim` accordingly. This would break the existing txt2img pipeline.

Option A is preferred because it doesn't change the transformer interface.

The projection layer would be a new component in the encoder:
```rust
quant_conv: Linear(64 * 2, 4)  // maps distribution params to 4-channel latent
```

This needs its own weight tensor — check if the checkpoint has a matching key. If not, initialize randomly (it would need training).

**Actually** — looking at the checkpoint more carefully, there may NOT be a `quant_conv`. In that case, the encoder's `conv_out` might directly output `2 * latent_channels` where `latent_channels = 4`, giving conv_out output of 8 or 9 (not 129). The 129 might come from a different interpretation. **This needs verification during implementation** by checking if any checkpoint key maps to a 64→4 projection.

### 4. Also fix `VideoDecoder` (bonus, needed for full pipeline)

The decoder has its own mismatch (README: "7 up_blocks vs 4 in Rust"). The Python decoder has:
- 7 up_blocks with timestep conditioning (`time_embedder` + `scale_shift_table` per resblock stage)
- `conv_in`: 128 → 1024
- `conv_out`: 128 → 48 (→ depth_to_space(r=4) → 3 RGB channels)
- `last_time_embedder` + `last_scale_shift_table`
- `timestep_scale_multiplier` (scalar)

This is a larger scope item. For now, **focus on the encoder only** — the decoder mismatch is a separate task documented in README. Without a working decoder, we can still do img2img (encode → denoise → save latent frames as PGM).
